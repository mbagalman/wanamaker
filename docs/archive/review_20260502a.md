## Findings

### P2: Terminology guardrail misses forecast copy

`tests/unit/test_terminology_guardrails.py` scans Jinja2 templates plus user-facing Python modules under `src/wanamaker/reports/` and `src/wanamaker/trust_card/`, but forecast modules also emit literal user-facing interpretation and explanation strings.

Affected examples:

- `src/wanamaker/forecast/scenarios.py` emits scenario interpretation strings that flow into CLI output and saved Markdown.
- `src/wanamaker/forecast/ramp.py` emits ramp recommendation explanations that flow into CLI output and saved Markdown.

Concern: future optimizer-language regressions in forecast copy, such as "optimized budget", "optimal allocation", "best budget", "guaranteed lift", or "maximize ROI" used as a promise, would not be caught by the guardrail despite the AGENTS.md instruction to add new user-facing string emitters to `USER_FACING_COPY_DIRS`.

Suggested fix: extend `USER_FACING_COPY_DIRS` to include the forecast copy emitters, or add a targeted allowlisted scan for `src/wanamaker/forecast/scenarios.py`, `src/wanamaker/forecast/ramp.py`, and any future forecast modules that emit rendered user-facing text.

**Resolution (2026-05-02):** Adopted a per-instance approach instead of extending `USER_FACING_COPY_DIRS`. Reason: both `forecast/scenarios.py` and `forecast/ramp.py` legitimately quote the banned phrases in their *module docstrings* when explaining what Wanamaker is *not* (e.g. `not a continuous "optimized budget."`). A whole-file static scan would produce false-positive failures on those defensive mentions. Coverage now stands at:

- `tests/unit/test_compare_scenarios.py::TestNoBannedTerminology` — already existed; covers `_interpretation_sentence` under multiple scenarios.
- `tests/unit/test_ramp.py::TestNoBannedTerminology` — added; covers `_explain_recommendation` and `_explain_no_recommendation` under every verdict (proceed / stage / test_first / do_not_recommend, with and without weak Trust Card, plus the spend-invariant short-circuit). 6 new tests.
- `tests/unit/test_terminology_guardrails.py` — module docstring updated to explicitly state that `forecast/` is gated per-instance and *why*, with a contributor-facing rule: when adding a new forecast user-facing string emitter, add a matching per-instance test rather than extending `USER_FACING_COPY_DIRS`.

### P3: Strict docs build currently fails on repo links

`mkdocs build --strict` aborts with 149 warnings, mostly because `docs/verification.md` links to repo files outside the docs tree, such as `../tests/...` and `../src/...`.

Concern: if strict docs builds are expected to gate releases, these verification links currently make the docs build fail even though the linked files exist in the repository.

Suggested fix: decide whether `docs/verification.md` should use MkDocs-compatible links, be excluded from strict link validation, or be treated as an internal audit artifact outside the published docs nav.

**Resolution (2026-05-02):** Took the MkDocs-compatible-links route. All 150 links in `docs/verification.md` that pointed at repo-internal source / test / config files (`../src/...`, `../tests/...`, `../pyproject.toml`, `../mkdocs.yml`, `../benchmark_data/`) were rewritten to absolute GitHub URLs (`https://github.com/mbagalman/wanamaker/blob/master/...` for files, `tree/master/...` for directories). Links to other docs under `docs/` remain relative. The same fix was applied to two equivalent links in `docs/references/risk_adjusted_allocation_math.md`. `mkdocs build --strict` now completes with **zero warnings** (down from 149); the only remaining INFOs are about archived files (`archive/feedback.md`, this review file) being intentionally absent from the nav.

Trade-off: the absolute URLs hardcode the repo URL and branch name. `mkdocs.yml` already hardcodes the same `repo_url`, so this is not a new failure mode; if the repo ever moves, a single `sed` updates both surfaces.
